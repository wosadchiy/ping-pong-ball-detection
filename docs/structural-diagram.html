BallTrackerPro — Структурная схема САУ

Схема для воссоздания текущей системы управления в MATLAB/Simulink. Показывает полный тракт «ball → camera → Python → USB → Arduino → motor → camera mount → ball-in-frame» с разделением: что считается на хосте, что во встроенном контроллере, что — физика. Сейчас это разомкнутый следящий привод: внешний оптический контур замкнут через камеру, а внутренний контур двигателя полностью открыт — ни датчика положения, ни тахогенератора, ни датчика тока (см. красную пунктирную область на схеме).

Связанные документы: control-system.html — диаграмма потока данных по файлам; drive-simulator.html — браузерный симулятор драйвера (Timer2 + vel_table) с тремя графиками; closed-loop-sim.html — браузерный симулятор замкнутого контура (синусоидальный вход, |H(jω)|, фазовый запас) — каждый блок этой схемы там численно реализован.

1. Структурная схема (текущее состояние)

Python host (CV, фильтрация, формирование задания) Arduino MCU (parser, П-регулятор, ramp, step driver) Физика (механика, оптика, ball-в-кадре) Визуальная обратная связь (единственная замкнутая)

2. Полная задержка системы (latency budget)

Это время от «мяч сместился в реальности» до «мотор начал отрабатывать смещение». Складывается из последовательных задержек по тракту прямого хода. Указаны чистые задержки (transport delay) и эквивалентные задержки фильтров первого порядка.

#Звено Источник τ @120 FPS / 150 Hz logic τ @60 FPS / 90 Hz logic
1 Камера (экспозиция + USB-кадр) 1 / FPS_cam ~ 8 мс ~ 17 мс
2 OpenCV: blur+HSV+morph+contour+circle профилировано на host ~ 5 мс ~ 8 мс
3 EMA-фильтр (эквив. T_ф = T_s·(1−α)/α) α=0.25, T_s=1/FPS_logic ~ 20 мс ~ 33 мс
4 Serial TX (30 байт ÷ 11520 байт/с + jitter) UART 115200 baud ~ 4 мс ~ 4 мс
5 Arduino loop() + parser измерено на DEBUG_RX_PULSE D8 ~ 1 мс ~ 1 мс
6 Timer2 ramp до новой ω (одна ISR-итерация) 1 кГц tick ~ 1 мс ~ 1 мс
7 Timer1 → импульс на драйвер (HW PWM) аппаратный, без джиттера ≪ 1 мс ≪ 1 мс
8 Механика мотора (нарастание тока в обмотке) L/R обмотки шагового ~ 1–2 мс ~ 1–2 мс
Σ полная задержка (от мяча до момента на валу): ≈ 40 мс ≈ 65 мс
+ дополнительный «лаг разгона» поверх этого бюджета: при больших командных скачках ω (например, мяч резко сдвинулся к краю кадра) Timer2 ramp ещё не сразу доводит мотор до целевой скорости — ограничение α = 500 user/с² означает, что от 0 до ω_max=100 уходит дополнительно 100/500 = 200 мс. Это не задержка в смысле transport delay (фаза не сдвигается на эту величину для малых синусоидальных возмущений), но она режет полосу пропускания контура и проявляется как дополнительное запаздывание на больших шагах.
Что значит «отставание красного крестика», которое ты видишь:
Красный крестик визуализирует (f_nx, f_ny) — выход EMA. То есть он показывает именно задержку звена № 3 (плюс маленький вклад от №1+2). Лаг звеньев 4–8 крестиком не виден — он влияет только на поведение мотора. Поэтому когда жёлтый круг (детекция) и красный крестик (EMA) расходятся на 5–10 px при движении мяча — это ровно фазовое запаздывание EMA на текущей частоте мяча.

Запас по фазе и предел K_p

На частоте gain crossover ω_c (где |L(jω)|=1) запас по фазе:

φ_m ≈ 90° − arctan(ω_c · T_ф) − ω_c · τ_pure_delay

τ_pure_delay = τ_кам + τ_дет + τ_тх + τ_ard ≈ 18 мс @120 FPS
T_ф ≈ 20 мс @150 Hz logic

При K_p = 0.45 расчётная ω_c ≈ 12–15 рад/с, что даёт φ_m ≈ 25–35° — на грани классического «запаса 30–45°». Это и есть экспериментально наблюдаемый предел устойчивости.
Главный вывод по задержкам: 50–60 % всей задержки сидит в EMA (звено №3). Уменьшение её — самый дешёвый способ поднять ω_c и следовательно K_p. Камера + USB + Arduino дают суммарно всего ~15 мс (на Windows@120 FPS), их трогать не надо.

4. Передаточные функции (готовы для MATLAB/Simulink)

Для непрерывной аппроксимации (прототипирование в s-области). Все «z-блоки» ниже сначала записаны дискретно (как в коде), потом — их непрерывный эквивалент через c2d/d2c. Для шаговика предполагается идеальное преобразование «команда ω → угол» без потери шагов — так система получается чисто кинематической.

W_опт(s) = K_л  статическая, 539 px/рад

K_л = (w/2) / (FOV_x/2) = 320 / (34·π/180)
≈ 539.0 px/рад
W_дет(s) ≈ K_д · e^(−τ_д·s) · ZOH(T_к)

K_д = 1 (камера+OpenCV выдают пиксели 1:1)
τ_д ≈ 5–10 мс (HSV+morph+contour)
T_к = 1/FPS_cam ≈ 11–17 мс (квантование)
W_ф(z) = α / (1 − (1−α)·z⁻¹)

α = 0.25, T_s = 1/FPS_logic
Эквивалент: W_ф(s) ≈ 1 / (T_ф·s + 1)
T_ф = T_s · (1−α)/α ≈ 27 мс @150 Hz
W_тх(s) = e^(−τ_тх·s)  чистое запаздывание

τ_тх ≈ 30 байт / 11520 (байт/с) = 2.6 мс
+ jitter ОС-планировщика → ≈ 3–4 мс
W_норм(s) = 1 / L  статическая

L = 320 px
err = nx / 320 ∈ [−1; +1]
W_р(s) = K_p · ω_max  П-регулятор

K_p ∈ [0; 5], ω_max ∈ [0; 100]
ω_t = err · K_p · ω_max
W_sat: ω_t → ω_t' = clamp(ω_t, ±ω_max)

Жёсткое насыщение, не передаточная — нелинейность.
В Simulink: блок Saturation с пределами ±ω_max.
W_ramp: rate-limiter с производной ≤ α

В Simulink: блок Rate Limiter
Rising/Falling slew rate = ±α user/с²
α = ω_max·1000/(V_TABLE_N · accel_skip)
α ∈ [≈ 2; 500] user/с² при ω_max=100
W_pulse: ω → f_step = 89.5·|ω| [Hz]

Нелинейность: T = 11172/|ω| [мкс]
В Simulink: Lookup Table по vel_table[]
Для непр. аппроксимации: f_step = K_step·ω, K_step = 89.5
W_шд(s): идеальный шаговик

θ̇_вал = (2π/N_step) · f_step
N_step = 200 · μ_step (μ — микрошаг, ≈ 1/16 типично)
→ K_шд = (2π·89.5)/(200·μ_step) рад/с / user-unit
W_кин(s) = (1/i_red) · (1/s)

Интегратор: θ_cam = ∫θ̇_вал · dt / i_red
i_red — передаточное отношение (1 при прямом креплении)
Динамика инерции J опущена (шаговик «идеален»)
Замыкание контура (визуальное):

e_θ = θ_ball − θ_cam
px = K_л · e_θ → детектор → EMA → … → мотор → θ_cam → ⊕
Полный разомкнутый коэф. усиления на низких частотах:
K_loop = K_л · K_д · 1 · (1/L) · K_p · ω_max · K_шд · 1

Полный передаточный коэффициент (s-область, низкие ω)

W_open(s) = W_тх · W_ф · W_дет · W_опт   (хост)
       × (1/L) · K_p · ω_max · W_sat · W_ramp · W_pulse · W_шд · W_кин  (MCU+физика)

Без насыщения и при «небольшом» сигнале (W_sat=1, W_ramp=1):
W_open(s) ≈ K_loop · e^(−τ·s) / [(T_ф·s + 1) · s]
где τ = τ_д + τ_тх ≈ 8–14 мс, T_ф ≈ 27–67 мс (зависит от FPS),
K_loop = K_л · (1/L) · K_p · ω_max · (2π · 89.5 / (N_step · i_red)).

5. Что разомкнуто, что нужно добавить для классической трёхконтурной схемы

В твоём втором скриншоте показана типовая структура цифрового следящего привода с тремя вложенными контурами регуляторов. Сравним:

КонтурКлассика (скриншот)У нас сейчасЧто нужно
Тока (внутренний) W_рт + датчик K_от отсутствует — драйвер шаговика отрабатывает ток сам, в open-loop по фазам В шаговике с Step/Dir драйвером недоступно. Если перейти на BLDC + FOC — добавится автоматически.
Скорости W_рс + ТГ K_ос отсутствует — Arduino отправляет открытое задание ω, шаговик «обязан» выдержать Энкодер на валу + дифференцирование угла (либо тахогенератор) → внутренний ПИ-контур скорости
Положения W_рп + энкодер K_оп отсутствует локально на валу; есть «оптическое замещение» через камеру Абсолютный энкодер вала (или интеграл импульсов с поправкой на пропуски)
Слежения (внешний) — есть: камера → П-регулятор с большой задержкой (EMA + камера + USB) Это и есть наш «следящий контур». Запас по фазе ограничен суммой задержек выше.
Поэтому сейчас система — «следящий привод с разомкнутыми внутренними контурами»: единственная обратная связь — оптическая через камеру. Все динамические свойства мотора (инерция, упругость ремня, момент нагрузки) ничем не компенсируются — они напрямую подмешиваются в фазу внешнего контура.

Куда расти

  • Минимум для замкнутого скоростного контура: энкодер AS5048 или AMT102 на вал → второй UART/SPI на Arduino → ПИ по ошибке скорости → новая W_рс(s). Заменит «доверие шаговику».
  • D-член в текущем П-регуляторе: разовое решение для стабильности без датчиков (D от пиксельной ошибки). 1 час работы, +50% по K_p.
  • Velocity feed-forward: Python считает d(nx)/dt и отправляет 8-м полем; Arduino добавляет в команду ω. Снимает с П-контура задачу «догонять движущийся мяч».
  • Smith predictor: если в MATLAB модель окажется хорошей, можно компенсировать чистое запаздывание τ_д+τ_тх внутри хоста.

6. Подсказки для Simulink-модели

Discrete-time стек (точная репликация кода)

• Sample time T_s = 1/150 с (логический поток)
• Camera ZOH: Zero-Order Hold с T_cam = 1/60 c
• EMA: Discrete Filter num=[α], den=[1, α−1]
• τ delays: Transport Delay или Delay блок
• Saturation: Saturation
• Ramp: Rate Limiter (R↑ = R↓ = α)
• Pulse-period: 1-D Lookup Table по vel_table
Continuous-time упрощённая (для синтеза)

• Все блоки → Transfer Fcn
• EMA: 1/(T_ф·s+1), T_ф=0.027
• Камера: 1 (или 1/(T_cam·s+1) если хочешь)
• Запаздывания: Pade approximation 2-го порядка
• Шаговик: K_step / s (интегратор сразу до θ_cam)
• Регулятор: Gain × Saturation
• Можно использовать linmod + margin для получения границы устойчивости.
Гипотеза, которую интересно проверить в модели: при текущих параметрах (T_ф ≈ 27–67 мс, τ ≈ 8 мс, K_шд от микрошага) расчётная граница устойчивости по K_p должна выйти примерно на 0.4–0.6 — то что ты и видишь экспериментально. Если так — модель адекватна, и можно уверенно начинать пробовать D-член или прогноз на ней.
