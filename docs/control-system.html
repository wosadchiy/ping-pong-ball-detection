BallTrackerPro — Control System

Path of a single video frame from the camera sensor down to a step pulse on the motor coil. The system is split across two CPUs that talk over a 115 200 baud USB serial link: a Python host doing computer vision and parameter tuning, and an Arduino sketch doing real-time motor control with hardware-timed step pulses.

1. End-to-end data flow

Python host (computer vision, UI) Arduino firmware (real-time) Shared parameter (host → MCU)

2. Stage-by-stage breakdown

1 · Camera capture Python · capture thread

OpenCV grabs raw frames in BGR through the platform-native backend (AVFoundation on macOS, V4L2 on Linux, DShow on Windows). The capture thread only does cap.read(); all processing happens in the logic thread on the freshest available frame, so a slow detector never backpressures the camera.

camera.py · VideoStream

2 · Ball detection & smoothing Python · logic thread

Frame → GaussianBlur(11) → HSV → cv2.inRange using (h_min..h_max, s_min..255, v_min..255) → morph open/close (7×7 kernel) → contour pick by area > 500 px and circularity > 0.5 → minEnclosingCircle gives (cx, cy). The pixel error relative to frame centre is clamped to ±cx_f then EMA-smoothed with α = 0.25.

detector.py · BallDetector.process

3 · Serial TX Python · logic thread

A single line per logic iteration: "ax,ay,nx,ny,kp,tracking,max_omega\n". Floats with two decimals so the EMA's sub-pixel resolution survives down the wire. The write is non-blocking (timeout=0.001); dropped writes are swallowed silently — the watchdog on the MCU handles the silence.

hardware.py · ArduinoHandler.send_data

4 · Parser Arduino · loop()

Bytes accumulate into inputBuffer until '\n'; then parseIncomingData splits on six commas, parses with toFloat() / toInt(), and updates the live parameter set (normX, Kp, max_omega, isTracking). Only normX currently affects the motor — angleX/angleY and normY are received for diagnostics / future use.

cameraControl.ino · parseIncomingData

5 · Watchdog Arduino · loop()

Every successful parse stamps last_packet_time = millis(). A boolean python_active says whether a packet arrived in the last 250 ms. If host falls silent the motor command falls back to zero in the very next iteration — there is no catch-up logic, no last known good value being held; the goal is to fail safe.

6 · Mode select Arduino · loop()

A simple priority ladder:

  1. Left button pressed → ω_preset = −10 (manual jog)
  2. Right button pressed → ω_preset = +10
  3. Python active AND isTracking → use the P-control law (block 7)
  4. Else → ω_preset = 0

The buttons always win — they're a hard manual override.

7 · Normalized P-control Arduino · loop()

Pixel error is normalised against half the sensor width (SENSOR_HALF_WIDTH_PX = 320) so the gain Kp is independent of the capture resolution. With Kp = 1.0 a ball at the edge of the frame drives the motor at exactly max_omega; with Kp = 0.5 it runs at half. See block 3 below for the formula and the rationale.

8 · Acceleration smoother Arduino · loop()

Once per millisecond, the actual omega drifts toward omega_preset by ±0.15 units, snapping to the target when within 0.05. This prevents the stepper from being asked for instantaneous velocity changes — it would slip or stall. Effective acceleration is 150 units/s.

9 · Step pulse driver Arduino · Timer1

Steps are produced in hardware by Timer1 in CTC mode. The step period is OCR1A = 11172 / |ω| (clamped to ≥20 to keep below the driver's max step rate), so faster ω = shorter period = more pulses per second. Direction goes to dirPin, the EN pin is held LOW (driver enabled) while the motor is moving. Below |ω| < 0.1 the PWM output is detached and EN can release.

3. The control law

The whole loop reduces to two lines on the Arduino. All complexity sits in the parameters and in how normX is conditioned upstream.

err = normX / SENSOR_HALF_WIDTH_PX    // dimensionless, in [−1, +1]
ω_preset = err · max_omega · Kp   // units = motor speed (steps/sec internally)
ω_preset = clamp(ω_preset, −max_omega, +max_omega)

Why the normalisation matters

Earlier the firmware ran omega_preset = normX · Kp on raw pixels. Because normX spans ±320, even a modest Kp = 1.0 meant ω = 320 at the edge — far above any reasonable max_omega (currently 40). To keep the motor sane the user had to pick Kp ≈ 0.03–0.04, which made the slider feel useless.

Normalising the error to [−1, +1] decouples three knobs that were tangled into a single multiplication:

KnobOwned byMeans
SENSOR_HALF_WIDTH_PXArduino constantSensor scale (depends on capture width)
max_omegaPython slider → serialPhysical speed cap (steps/sec)
KpPython slider → serialUser aggressiveness, 0–1 typical
Result: Kp = 1.0 is always “motor saturates at max_omega when the ball reaches the screen edge” regardless of capture resolution or chosen speed cap. Kp acts purely as user gain and Kp > 1.0 means intentional over-driving — the clamp catches it.

5. Timing & safety

Camera

60–90 FPS depending on driver / exposure setting. Independent of the logic thread.

Logic loop

~150–200 Hz on a modern host (it processes the latest frame; the same frame is processed multiple times if the camera lags).

Serial

One packet per logic iteration. At 115 200 baud and ~30-byte packets the channel can sustain ~400 packets/s — never the bottleneck.

Arduino loop()

Runs as fast as possible (a few kHz). Step pulses themselves are generated by Timer1 in hardware, so loop() jitter doesn't ruin them.

Smoother

Time-quantised to 1 ms via millis(). Effective acceleration: 150 units/s → from 0 to max_omega = 40 in ~270 ms.

Watchdog

250 ms silence from host → automatic stop. USB unplug, host crash, Python exception, all collapse to "ω_preset = 0" with no special handling.

Failure modes the loop is intentionally protected against:
  • Host crash mid-recording → watchdog stops the motor in 250 ms.
  • Detector noise glitch → EMA smoother absorbs it; pixel-clamp prevents a runaway from a misdetected contour off-screen.
  • User asks for impossible speed (Kp ≫ 1) → clamp at max_omega.
  • Manual override needed → buttonPin1/2 always win, even with a packet flood from Python.
